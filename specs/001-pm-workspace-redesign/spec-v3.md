# Feature Specification: PM 工作台 v3 — 版本跟踪·编辑器定制·批注·溯源增强

**Feature Branch**: `002-pm-v3-enhancement`

**Created**: 2026-06-30

**Status**: Draft

**Input**: User description: "1.当前所有模块的内容新增都要显示版本跟踪是哪个版本内容；需要支持筛选；分页；排序；溯源功能也要对其进行跟踪 3.定制化一下PM工作台的富文本编辑器；都需要内置自己的版本更迭不同于外面的版本，每次更新；手动保存用户确认后产生新的版本，需要有时间戳；自动保存的版本号不变。对比功能：并排展示对应的内容。红绿展示增减行；合并支持每行、多行选择的加入到现有版本；新增功能：批注功能，用户选择对应内容，并弹出功能选择批注按钮，用户输入批注内容，标黄展示批注。加入ai功能可以一键对对应功能进行ai修改并插入。 4.当前很多功能都做了调整，和PRD文档偏差较大，需要同步一下整体功能并test校验，溯源，展示等能力。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 条目列表筛选/排序/分页 + 版本跟踪直显 (Priority: P1)

作为产品经理，我希望在任意模块的条目列表中按版本、状态、优先级筛选，支持排序和分页，并且每个条目都清晰显示其版本归属。

**Why this priority**: 当前列表仅有文本搜索，无法按业务维度筛选；版本徽章仅显示在卡片视图，表格视图缺失版本列。

**Independent Test**: 打开任意模块，使用筛选栏按版本号/状态/优先级过滤，验证排序切换和分页导航正常工作。

**Acceptance Scenarios**:

1. **Given** 用户打开任意模块条目列表，**When** 页面加载完成，**Then** 搜索栏下方显示筛选条（版本号、状态、优先级）
2. **Given** 用户选择状态筛选为"草稿"，**When** 应用筛选，**Then** 列表仅显示状态为草稿的条目
3. **Given** 条目列表超过 20 条，**When** 滚动到底部，**Then** 显示分页导航，可切换页码
4. **Given** 用户点击"更新时间"列头，**When** 切换排序，**Then** 列表按更新时间升降序重排
5. **Given** 用户查看表格视图，**When** 查看行内容，**Then** 每行显示版本徽章列
6. **Given** 用户在溯源图中查看节点，**When** 节点有版本信息，**Then** 节点上显示版本徽章

---

### User Story 2 - 编辑器版本控制：手动保存产版本 + 自动保存不变版本 (Priority: P1)

作为产品经理，我希望富文本编辑器有独立的版本机制：手动保存时确认是否产生新版本（含时间戳），自动保存仅更新内容不递增版本号。

**Why this priority**: 版本管理是 PM 文档工作的核心需求，区分手动/自动保存让版本历史清晰可控。

**Independent Test**: 打开 PRD 文档编辑器，编辑内容后等待自动保存（版本号不变），手动保存并确认新版本（版本号递增）。

**Acceptance Scenarios**:

1. **Given** 用户在编辑器中修改内容，**When** 30 秒无操作，**Then** 自动保存触发，版本号不变，显示"自动保存于 HH:MM"
2. **Given** 用户点击"保存"按钮，**When** 弹出确认框，**Then** 用户可选择"创建新版本"或"仅保存内容"
3. **Given** 用户选择"创建新版本"，**When** 保存完成，**Then** 版本号递增，EntryVersion 记录含时间戳
4. **Given** 用户选择"仅保存内容"，**When** 保存完成，**Then** 内容更新但版本号不变
5. **Given** 编辑器头部区域，**When** 查看状态，**Then** 显示当前版本号 + 保存状态

---

### User Story 3 - 版本比较：并排展示 + 红绿 diff (Priority: P1)

作为产品经理，我希望并排查看两个版本的内容差异，增加行标绿、删除行标红，行号对齐且同步滚动。

**Why this priority**: 版本比较是判断变更内容的核心工具，红绿 diff 是业界标准。

**Independent Test**: 打开文档编辑页，点击"版本比较"，选择两个版本，验证左右并排展示、红绿高亮、行号对齐。

**Acceptance Scenarios**:

1. **Given** 用户在文档编辑页，**When** 点击"版本比较"按钮，**Then** 展开左右并排面板
2. **Given** 版本比较面板已展示，**When** 查看差异，**Then** 增加行绿色背景 + gutter 显示"+"
3. **Given** 版本比较面板已展示，**When** 查看差异，**Then** 删除行红色背景 + gutter 显示"-"
4. **Given** 版本比较面板已展示，**When** 查看未变更内容，**Then** 正常显示 + 行号
5. **Given** 用户滚动左侧面板，**When** 滚动时，**Then** 右侧面板同步滚动

---

### User Story 4 - 版本合并：行级/多行选择 (Priority: P1)

作为产品经理，我希望在合并分支时能逐行或批量选择采用哪个版本的变更。

**Why this priority**: 分支并行开发后需要合并回主线，行级选择让冲突解决精细可控。

**Independent Test**: 在分支上编辑后点击"合并到主线"，验证冲突行高亮、逐行选择、多行批量选择功能。

**Acceptance Scenarios**:

1. **Given** 用户查看合并面板，**When** 存在冲突行，**Then** 冲突行高亮显示
2. **Given** 用户查看冲突行，**When** 点击"采用源"或"采用目标"，**Then** 该行内容更新为选择方
3. **Given** 用户需要批量选择，**When** 勾选多行，**Then** 批量应用选择
4. **Given** 用户解决所有冲突，**When** 点击"完成合并"，**Then** 创建新 EntryVersion（branchName='main'）

---

### User Story 5 - 批注功能：选中文本标黄批注 + AI 修改 (Priority: P1)

作为产品经理，我希望选中文本后添加批注（标黄高亮），并支持一键 AI 修改批注内容。

**Why this priority**: 批注是文档评审的核心交互，AI 辅助修改提升修改效率。

**Independent Test**: 在编辑器中选中文本，添加批注验证标黄高亮和批注列表；点击 AI 修改验证内容替换。

**Acceptance Scenarios**:

1. **Given** 用户在编辑器中选中文本，**When** 出现浮动工具栏，**Then** 工具栏包含"批注"按钮
2. **Given** 用户点击"批注"按钮，**When** 弹出输入框，**Then** 用户输入批注内容后，选中区域标黄高亮
3. **Given** 文档中有批注，**When** 查看右侧批注面板，**Then** 显示所有批注列表，点击跳转定位
4. **Given** 用户在批注面板点击"AI 修改"，**When** AI 返回结果，**Then** 修改内容替换选中文本

### User Story 6 - 溯源增强：版本感知 + 筛选 (Priority: P2)

作为产品经理，我希望溯源图中节点显示版本信息，连线创建时记录版本快照，并可按模块/版本/关系类型筛选。

**Acceptance Scenarios**:

1. **Given** 用户在溯源图查看节点，**When** 条目有版本信息，**Then** 节点上显示版本徽章
2. **Given** 用户在溯源图创建连线，**When** 关联创建成功，**Then** 连线记录当前版本快照
3. **Given** 用户使用溯源图筛选栏，**When** 按模块类型/版本号/关系类型筛选，**Then** 图中仅显示匹配的节点和连线

### User Story 7 - PRD 功能同步 & 整体校验 (Priority: P2)

作为产品经理，我希望对照 PRD 文档全面校验已实现功能，确保溯源、版本、展示等能力与设计一致。

**Acceptance Scenarios**:

1. **Given** 对照 spec-v2 的 FR-001 到 FR-027，**When** 逐项校验，**Then** 标记已实现/部分实现/未实现并修复差距
2. **Given** 溯源图连线创建，**When** 验证 API 调用流程，**Then** 确认 Relation 正确创建

### Edge Cases

- 自动保存与手动保存同时触发时，手动保存优先，取消 pending 自动保存定时器
- 版本比较选择同一版本时，提示"请选择不同版本"
- 批注选区与其他格式化标记交叉时，批注高亮不破坏已有格式
- 合并面板无冲突时，自动合并并提示
- 筛选条件组合无结果时，显示空状态提示

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 所有模块条目列表 MUST 支持按版本号、状态、优先级筛选
- **FR-002**: 所有模块条目列表 MUST 支持按更新时间、创建时间、优先级、版本号排序（升降切换）
- **FR-003**: 所有模块条目列表 MUST 支持分页（每页 20 条）
- **FR-004**: 表格视图行 MUST 显示版本徽章列
- **FR-005**: 编辑器 MUST 区分手动保存和自动保存：自动保存不递增版本号，手动保存弹出版本确认
- **FR-006**: 手动保存确认创建新版本时 MUST 记录时间戳（ISO 8601）
- **FR-007**: 编辑器头部 MUST 显示当前版本号和保存状态
- **FR-008**: 版本比较 MUST 以并排双栏展示，左侧旧版、右侧新版
- **FR-009**: 版本比较 MUST 用绿色背景标记增加行、红色背景标记删除行
- **FR-010**: 版本比较 MUST 支持左右同步滚动和行号对齐
- **FR-011**: 版本合并 MUST 支持逐行选择"采用源"/"采用目标"
- **FR-012**: 版本合并 MUST 支持多行批量选择
- **FR-013**: 版本合并完成后 MUST 创建新 EntryVersion（branchName='main'）
- **FR-014**: 编辑器 MUST 支持选中文本后添加批注（标黄高亮）
- **FR-015**: 批注面板 MUST 显示当前文档所有批注列表，点击跳转定位
- **FR-016**: 批注项 MUST 支持 AI 一键修改
- **FR-017**: 溯源图节点 MUST 显示版本徽章
- **FR-018**: 溯源连线创建时 MUST 记录当前版本快照
- **FR-019**: 溯源图 MUST 支持按模块类型、版本号、关系类型筛选
- **FR-020**: 全部功能 MUST 对照 PRD 文档校验一致性

### Key Entities

- **EntryAnnotation**: 批注 — entryId, entryVersionId, textRange, selectedText, content, highlightColor, createdBy, createdAt, updatedAt
- **Relation** (扩展): 增加 versionSnapshot 字段记录关联时的版本快照

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在 3 秒内通过筛选+排序定位到任意条目
- **SC-002**: 手动保存创建新版本的操作在 2 秒内完成
- **SC-003**: 版本比较面板加载和差异展示在 3 秒内完成
- **SC-004**: 合并冲突行解决操作响应时间 ≤ 500ms
- **SC-005**: 批注添加和高亮渲染在 500ms 内完成
- **SC-006**: AI 修改批注内容返回结果在 10 秒内
- **SC-007**: PRD 功能校验覆盖率 100%

---

## Assumptions

- 批注数据存储在 entry.data.annotations（JSON 字段），无需后端新增表
- 版本比较使用纯文本 diff（先 strip HTML 标签），后续可升级为 TipTap JSON 级别 diff
- AI 批注修改复用 OpenWebUI 已有的 chat API 通道
- 前端分页先基于全量加载 + 前端切片
- TipTap 批注扩展采用基础版（单层高亮+弹窗），不支持嵌套批注
