# Feature Specification: 产品架构页面缺陷修复与交互重构

**Feature Branch**: `[005-arch-bugfix-redesign]`

**Created**: 2026-07-09

**Status**: Draft

**Input**: User description: "产品架构页面存在多个操作失效 Bug 和交互设计问题：1) 操作按钮无法生效；2) 新增模块按钮点击后所有模块卡住；3) 添加参数无法正常新建，需改为批量新建表单；4) 创建版本列未显示版本信息；5) 关联需求列交互不清晰；6) 思维导图只显示产品架构一个节点，应显示项目→模块→功能层级；7) 移除版本选择器和 AI 按钮；8) 更新条目失败"

## Clarifications

### Session 2026-07-09

- Q: 批量新建表单应该支持哪些条目类型的批量创建？ → A: 仅参数支持批量新建，功能和模块保持单条新建
- Q: 应保留哪个表格组件作为主组件？ → A: 保留 ArchitectureTable，删除 ModuleTable 及其依赖的 architecture store
- Q: 批量新建表单提交时部分成功部分失败应如何处理？ → A: 全部成功或全部失败（all-or-nothing），任一失败则全部不创建，显示错误让用户修复后重试

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 修复操作失效与数据更新失败 (Priority: P1)

作为产品经理，我需要在表格视图中点击编辑、新增、删除等操作按钮时，系统能正确响应并完成数据变更，而不是操作无响应或报错失败。

**Why this priority**: 操作按钮完全失效是阻塞性 Bug，用户无法进行任何数据管理操作，页面功能形同虚设。必须首先修复。

**Independent Test**: 可以独立测试：在表格视图中依次点击编辑、新增下级、删除按钮，验证每个操作是否正确触发弹窗或执行 API 调用，且数据变更后页面正确刷新。

**Acceptance Scenarios**:

1. **Given** 用户在表格视图中点击某模块的"编辑"按钮，**When** 按钮被点击，**Then** 弹出编辑弹窗，表单预填当前模块数据，修改后保存能成功更新数据并刷新列表
2. **Given** 用户点击"新增模块"按钮，**When** 按钮被点击，**Then** 弹出新增模块弹窗，填写信息后保存能成功创建模块，且不影响其他已有模块的交互状态
3. **Given** 用户点击某功能的"添加参数"按钮，**When** 按钮被点击，**Then** 弹出新增参数弹窗，填写信息后保存能成功创建参数并关联到对应功能
4. **Given** 用户编辑某条目并提交，**When** API 调用完成，**Then** 系统显示"更新成功"提示，且表格数据即时刷新反映变更内容

---

### User Story 2 - 思维导图层级结构修正 (Priority: P1)

作为产品经理，我需要在思维导图视图中看到完整的项目→模块→功能三级树形结构，而不是仅显示"产品架构"一个根节点。

**Why this priority**: 思维导图是产品架构的核心可视化视图，当前仅显示一个节点完全无法传达架构信息，属于严重功能缺陷。

**Independent Test**: 可以独立测试：切换至思维导图视图，验证根节点显示项目名称，子节点显示模块，孙节点显示功能，且点击节点可查看详情。

**Acceptance Scenarios**:

1. **Given** 用户切换至思维导图视图，**When** 图形渲染完成，**Then** 根节点显示当前项目名称（而非固定的"产品架构"文字）
2. **Given** 思维导图已渲染，**When** 查看根节点的子节点，**Then** 每个模块显示为独立子节点，节点名称为模块名称
3. **Given** 某模块节点已展开，**When** 查看该模块的子节点，**Then** 每个功能显示为独立孙节点，节点名称为功能名称
4. **Given** 用户点击思维导图中的某个节点，**When** 节点被选中，**Then** 弹出详情弹窗，显示该节点的版本信息和描述

---

### User Story 3 - 批量新建参数表单 (Priority: P2)

作为产品经理，我需要通过批量新建表单一次性添加多个参数，而不是每次只能添加一条，以提高参数录入效率。功能和模块保持单条新建。

**Why this priority**: 批量新建是效率提升需求，不影响核心功能可用性，但能显著改善高频操作的用户体验。

**Independent Test**: 可以独立测试：点击"添加参数"按钮，验证弹窗支持添加多行参数数据，一次性提交后所有参数均成功创建。

**Acceptance Scenarios**:

1. **Given** 用户点击某功能的"添加参数"按钮，**When** 弹窗打开，**Then** 显示批量新建表单，默认包含 3 行空行，每行可填写参数名称、KEY、数据类型、是否必填
2. **Given** 用户在批量新建表单中填写了 3 行参数数据，**When** 点击"全部添加"按钮，**Then** 系统一次性创建 3 条参数记录，均关联到对应功能
3. **Given** 用户在批量新建表单中填写了部分行，**When** 点击"添加更多行"按钮，**Then** 表单底部新增一行空行供继续填写
4. **Given** 用户在批量新建表单中某行未填写名称，**When** 点击提交，**Then** 系统跳过未填写的空行，仅创建已完整填写的行

---

### User Story 4 - 版本信息与需求关联展示修正 (Priority: P2)

作为产品经理，我需要在表格视图中看到正确的版本信息和清晰的需求关联交互，而不是版本列为空或需求关联操作不明确。

**Why this priority**: 版本信息和需求关联是 spec 004 已定义的核心功能，当前实现存在数据展示和交互缺陷，需修正以匹配规格要求。

**Independent Test**: 可以独立测试：查看表格中"创建版本"列是否显示版本号，点击"关联需求"列是否能正确打开关联弹窗。

**Acceptance Scenarios**:

1. **Given** 某模块已创建且绑定了创建版本，**When** 用户查看表格中该模块的"创建版本"列，**Then** 列中显示版本号字符串（如"1.0.0"），而非空白或"-"
2. **Given** 某功能已关联 2 条需求，**When** 用户查看表格中该功能的"关联需求"列，**Then** 列中显示 2 个需求标签，每个标签显示需求名称
3. **Given** 用户点击某条目的需求标签或关联需求区域，**When** 点击操作触发，**Then** 弹出需求关联管理弹窗，支持新增绑定、单独解绑、批量清空

---

### User Story 5 - 移除不必要的顶部按钮 (Priority: P3)

作为产品经理，我希望移除页面顶部的"选择版本"和"AI"按钮，因为这些功能当前不需要或尚未实现，避免用户困惑。

**Why this priority**: 移除无用 UI 元素是清理性工作，不影响核心功能，优先级最低。

**Independent Test**: 可以独立测试：进入产品架构页面，验证顶部不再显示"选择版本"和"AI"按钮。

**Acceptance Scenarios**:

1. **Given** 用户进入产品架构页面，**When** 查看页面顶部操作区域，**Then** 不显示"选择版本"按钮和"AI"按钮
2. **Given** 版本选择器和 AI 按钮已移除，**When** 用户查看页面头部，**Then** 仅显示页面标题"产品架构"和思维导图/表格切换 Tab

---

### Edge Cases

- 当批量新建表单中所有行均为空时，提交按钮应禁用或提示"请至少填写一行数据"
- 当批量新建表单提交时部分参数创建失败，系统应全部回滚（不创建任何参数），显示错误信息让用户修复后重试
- 当思维导图数据为空（无模块无功能）时，应显示空状态提示而非空白画布
- 当版本信息字段在数据源中缺失时，"创建版本"列应显示默认值"1.0.0"而非空白
- 当 API 调用失败时，操作按钮应恢复可点击状态，不应导致整个页面卡死
- 当两个组件（ModuleTable 和 ArchitectureTable）同时存在时，需确认使用哪个组件作为主表格，避免数据流混乱

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 修复表格视图中所有操作按钮（编辑、新增、删除、复制）的点击响应，确保按钮点击后正确触发弹窗或 API 调用
- **FR-002**: 系统 MUST 修复"新增模块"按钮点击后导致所有模块交互卡死的 Bug，确保新增操作不影响其他模块的展开/收起/编辑等交互
- **FR-003**: 系统 MUST 修复"添加参数"按钮无法正常新建参数的 Bug，确保参数创建流程完整可用
- **FR-004**: 系统 MUST 将参数的单条新建表单改为批量新建表单，支持一次添加多条参数，表单默认包含 3 行空行，支持动态增减行数；功能和模块保持单条新建
- **FR-005**: 系统 MUST 修复"创建版本"列不显示版本信息的问题，确保该列正确展示条目的 `create_version` 值
- **FR-006**: 系统 MUST 修复"关联需求"列的交互问题，确保点击需求标签或关联区域时能正确打开需求关联管理弹窗
- **FR-007**: 系统 MUST 修正思维导图的层级结构：根节点显示项目名称，子节点显示模块，孙节点显示功能，节点点击后展示版本和描述信息
- **FR-008**: 系统 MUST 移除页面顶部的"选择版本"按钮和"AI"按钮
- **FR-009**: 系统 MUST 修复编辑条目后更新失败的问题，确保 API 调用成功后数据正确刷新
- **FR-010**: 系统 MUST 统一数据流，保留 ArchitectureTable 作为唯一表格组件，删除 ModuleTable 及其依赖的 architecture store（`$lib/stores/pm/architecture`），消除数据源冲突

### Key Entities *(include if feature involves data)*

- **ArchModule（模块）**: 产品架构一级条目，需确保 `createVersion`、`versionRecord`、`demandRelation` 字段在表格和思维导图中正确展示
- **ArchFeature（功能）**: 产品架构二级条目，需确保在思维导图中作为模块的子节点正确渲染
- **ArchParameter（参数）**: 产品架构三级条目，需支持批量新建，确保创建后正确关联到父功能
- **Project（项目）**: 思维导图根节点，需显示项目名称而非固定文字"产品架构"

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户点击表格中任意操作按钮后，系统在 200ms 内响应（弹出弹窗或执行操作），无卡死或无响应现象
- **SC-002**: 用户通过批量新建表单一次添加 5 条参数，全部在 3 秒内创建完成并正确显示在表格中
- **SC-003**: 思维导图视图正确显示项目→模块→功能三级结构，100% 的已有模块和功能均作为节点渲染
- **SC-004**: 表格"创建版本"列 100% 显示版本号字符串，无空白或"-"的情况（对于有版本数据的条目）
- **SC-005**: 编辑条目后保存成功率 ≥ 99%，保存后表格数据在 1 秒内刷新反映变更
- **SC-006**: 页面顶部不再显示"选择版本"和"AI"按钮，页面布局简洁无冗余元素

## Assumptions

- 当前存在两套表格组件（ModuleTable 和 ArchitectureTable），统一为 ArchitectureTable 作为主表格组件（ArchitectureTable 使用 architectureStore 的规范化数据流），删除 ModuleTable 及其依赖的 architecture store
- 思维导图根节点的项目名称可通过当前路由的 projectId 从项目信息 API 获取
- 批量新建表单的提交方式为逐条调用创建 API（而非批量 API），前端负责串行/并行调度
- 版本信息字段 `createVersion` 在 architectureStore 的 `convertToArchModules` 函数中未正确映射，需修复数据转换逻辑
- 操作按钮失效的根因可能是事件冒泡被阻止、组件状态管理冲突、或 API 调用异常未正确处理
- "新增模块"卡死的根因可能是 `modules = [...modules]` 触发了无限响应循环或状态更新冲突
