# Feature Specification: 产品架构模块版本溯源与需求关联

**Feature Branch**: `[004-arch-module-version-demand]`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "模块、功能也要支持关联需求稳定和版本信息；操作等。产品架构模块完整迭代 PRD（模块 / 功能 / 参数全层级统一版本溯源 + 需求关联）"

## Clarifications

### Session 2026-07-08

- **Q**: 版本履历存储方式 → **A**: Option A - `version_record` 作为内嵌 JSON 数组存储在模块/功能/参数表中，查询时直接读取
- **Q**: 版本履历"复制溯源日志"行为 → **A**: 导出为 CSV 格式 - 用户明确"整体应该是导出"，将"复制"改为"导出"功能，支持 CSV 文件下载
- **Q**: 并发编辑冲突处理策略 → **A**: 单用户工作台，不存在多用户并发编辑场景，无需乐观锁/悲观锁机制。版本履历仅记录当前用户的操作历史
- **Q**: 软删除后数据留存策略 → **A**: Option A - 保留完整数据，仅标记 `is_deleted`，版本履历和关联需求完整留存，管理员可在归档页面查看和检索
- **Q**: 版本履历数据量过大时的展示边界 → **A**: Option B - 浮窗展示最近 20 条记录，底部显示"查看更多"按钮，点击后展开完整列表或跳转到详情页面

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 模块/功能/参数统一版本溯源 (Priority: P1)

作为产品经理，我需要在创建或编辑模块、功能、参数时，系统能够自动记录每个条目的创建版本和全生命周期变更履历，且父子条目版本相互独立，互不影响。

**Why this priority**: 版本溯源是产品架构管理的核心能力，没有版本记录就无法追踪变更历史，是后续所有功能的基础。

**Independent Test**: 可以独立测试：创建模块/功能/参数后，检查每个条目是否自动绑定了创建版本，编辑后是否生成了版本变更记录，父条目修改不影响子条目版本。

**Acceptance Scenarios**:

1. **Given** 用户在工作台创建新模块，**When** 填写模块名称并保存，**Then** 系统自动将该模块的 `create_version` 设置为当前工作台版本，且不可手动修改
2. **Given** 用户编辑已存在的功能条目，**When** 修改功能描述并提交，**Then** 系统在当前环境版本下自动生成一条版本变更记录，追加到该功能的 `version_record` 履历中
3. **Given** 用户修改了某模块的版本履历，**When** 查看该模块下功能或参数的版本履历，**Then** 子条目版本履历保持独立，不受父模块变更影响

---

### User Story 2 - 条目与需求的多对多关联管理 (Priority: P1)

作为产品经理，我需要将模块、功能、参数与多条需求文档进行关联，支持新增绑定、单独解绑、批量清空，且已绑定需求以标签形式展示。

**Why this priority**: 需求关联是连接产品架构与需求管理的关键桥梁，直接影响产品迭代的可追溯性。

**Independent Test**: 可以独立测试：在表格或弹窗中为条目绑定/解绑需求，检查需求标签是否正确展示，解绑后是否生效。

**Acceptance Scenarios**:

1. **Given** 用户在表格视图中点击某模块的"关联需求"单元格，**When** 在弹窗中手动录入需求编号和文档链接并保存，**Then** 该需求以标签形式显示在单元格中，点击标签可跳转文档
2. **Given** 某功能已绑定 3 条需求，**When** 用户在编辑弹窗中解绑其中 1 条并保存，**Then** 该功能仅保留剩余 2 条需求关联
3. **Given** 用户选择清空某参数的关联需求，**When** 点击"批量清空"并确认，**Then** 该参数的 `demand_relation` 变为空列表，但历史版本履历中仍保留原始记录

---

### User Story 3 - 表格视图改版与统一操作 (Priority: P1)

作为用户，我需要在表格视图中看到包含"创建版本""版本履历""关联需求""操作"等统一列的完整信息，且三级条目操作权限一致。

**Why this priority**: 表格视图是用户日常操作的主要界面，视图改版直接影响使用体验。

**Independent Test**: 可以独立测试：打开表格视图，检查列顺序、版本履历悬浮浮窗、操作按钮是否按规格展示。

**Acceptance Scenarios**:

1. **Given** 用户进入产品架构模块的表格视图，**When** 查看表头，**Then** 列顺序为：名称 → 类型/KEY → 数据类型 → 必填 → 创建版本 → 版本履历 → 关联需求 → 描述 → 操作
2. **Given** 用户鼠标悬浮在某条目的"版本履历"单元格上，**When** 停留超过 300ms，**Then** 弹出浮窗显示：初始创建信息 + 历次修改列表（版本号/操作人/时间/变更字段），并支持复制整条溯源日志
3. **Given** 用户在操作列点击"新增下级"，**When** 当前条目为模块，**Then** 弹出新增功能弹窗；为功能时弹出新增参数弹窗；为参数时无"新增下级"按钮

---

### User Story 4 - 思维导图只读浏览与节点详情 (Priority: P2)

作为用户，我需要在思维导图视图中以只读方式浏览模块-功能-参数的树形结构，点击节点可查看详情弹窗。

**Why this priority**: 思维导图提供宏观视角，帮助理解产品架构全貌，但属于增强体验而非核心阻塞功能。

**Independent Test**: 可以独立测试：切换至思维导图视图，检查节点渲染、点击弹窗、交互限制是否符合只读约束。

**Acceptance Scenarios**:

1. **Given** 用户切换至思维导图视图，**When** 尝试拖拽节点或编辑节点文字，**Then** 操作被禁止，界面无任何变化
2. **Given** 用户点击思维导图中的某个模块节点，**When** 节点被选中，**Then** 弹出详情弹窗，显示：基础元信息、版本溯源板块、关联需求板块、操作按钮（跳转表格定位、复制条目信息）
3. **Given** 用户在思维导图视图中，**When** 查看顶部 Tab，**Then** 仅显示"思维导图/表格"切换，无版本选择按钮

---

### User Story 5 - 存量数据兼容与迁移 (Priority: P2)

作为系统管理员，我需要历史已存在的模块/功能/参数数据能够平滑迁移，自动填充默认版本信息。

**Why this priority**: 存量数据兼容确保升级过程中现有数据不丢失，但可以在核心功能完成后实施。

**Independent Test**: 可以独立测试：对存量数据执行迁移脚本，检查 `create_version` 是否填充为 `1.0.0`，`version_record` 和 `demand_relation` 是否正确初始化。

**Acceptance Scenarios**:

1. **Given** 系统中存在历史模块数据（无 `create_version` 字段），**When** 执行存量数据兼容迁移，**Then** 所有历史模块的 `create_version` 被批量填充为 `1.0.0`
2. **Given** 某历史功能条目在迁移前无版本履历，**When** 用户首次编辑该条目并保存，**Then** 系统自动抓取当前环境版本，生成第一条版本变更记录
3. **Given** 迁移后的历史参数条目，**When** 查看其关联需求，**Then** 默认显示为空列表，用户可手动补绑历史需求

---

### Edge Cases

- 当用户复制某条目时，若选择"同步复制已绑定需求"，新条目的需求关联生成新的独立关联记录（深拷贝），不引用原记录
- 当用户同时编辑某条目并修改其关联需求时，版本变更记录应包含所有被修改的字段名称及变更前后的值摘要
- 当工作台的当前上下文版本为空或不存在时，创建条目的 `create_version` 默认使用系统当前版本号（如从版本管理服务获取），若仍为空则使用默认值 `1.0.0`

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 为模块、功能、参数三类条目统一增加 `create_version`、`version_record`、`demand_relation` 三个字段
- **FR-002**: 系统 MUST 在新增条目时自动回填当前工作台上下文版本到 `create_version`，且该字段置灰只读
- **FR-003**: 系统 MUST 在编辑条目提交后，自动抓取当前环境版本，追加一条版本变更记录到 `version_record` 履历中
- **FR-004**: 系统 MUST 支持在表格视图和编辑弹窗中为条目绑定多条需求（`demand_id`、`demand_name`、`doc_link`）
- **FR-005**: 系统 MUST 支持单独解绑和批量清空已关联的需求
- **FR-006**: 系统 MUST 在表格视图中按固定顺序展示列：名称 → 类型/KEY → 数据类型 → 必填 → 创建版本 → 版本履历 → 关联需求 → 描述 → 操作
- **FR-007**: 系统 MUST 在"版本履历"列支持 hover 悬浮浮窗，展示初始创建信息和历次修改列表，并支持导出溯源日志为 CSV 格式
- **FR-015**: 系统 MUST 在版本履历浮窗中提供"导出"按钮，点击后生成 CSV 文件下载，包含所有版本变更记录的完整字段
- **FR-008**: 系统 MUST 在操作列统一提供：编辑、新增下级、删除、复制、查看溯源详情按钮，且根据条目类型控制"新增下级"按钮的显隐
- **FR-009**: 系统 MUST 在思维导图视图中以只读模式渲染模块-功能-参数树形结构，禁止拖拽、新增、删除、编辑、连线等操作
- **FR-010**: 系统 MUST 支持点击思维导图节点弹出统一详情弹窗，包含基础元信息、版本溯源、关联需求、操作按钮四个分区
- **FR-011**: 系统 MUST 在删除条目时执行软删除，保留版本履历和关联需求关系在后台可检索归档
- **FR-012**: 系统 MUST 在复制条目时生成全新的 `create_version`（不继承原条目履历），并支持选择是否同步复制已绑定需求
- **FR-013**: 系统 MUST 确保父子条目版本独立，修改父模块版本履历不会同步修改下级功能、参数数据
- **FR-014**: 系统 MUST 对存量数据执行兼容迁移：历史条目 `create_version` 默认填充 `1.0.0`，`version_record` 为空，首次编辑后生成第一条变更记录，`demand_relation` 默认空列表

### Key Entities *(include if feature involves data)*

- **Module（模块）**: 产品架构的一级条目，代表一个独立的产品模块。新增字段：`create_version`（字符串，创建时版本）、`version_record`（数组，变更履历）、`demand_relation`（数组，关联需求）。支持新增下级功能。
- **Function（功能）**: 产品架构的二级条目，隶属于某个模块。字段同模块。支持新增下级参数。
- **Parameter（参数）**: 产品架构的三级条目，隶属于某个功能。字段同模块。无下级新增按钮。
- **VersionRecord（版本履历项）**: 单条变更记录，包含：`version`（版本号）、`operator`（操作人）、`time`（操作时间）、`change_detail`（变更字段说明）。
- **DemandRelation（需求关联项）**: 单条需求关联，包含：`demand_id`（需求编号）、`demand_name`（需求名称）、`doc_link`（文档链接）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在 3 分钟内完成一个新模块的创建，且系统自动绑定的版本信息准确无误
- **SC-002**: 版本履历悬浮浮窗的加载时间不超过 500ms，即使单条条目有 50 条以上变更记录
- **SC-003**: 95% 的存量数据在迁移后能够正确显示默认版本信息（`create_version = 1.0.0`），无数据丢失
- **SC-004**: 用户可以在编辑弹窗中完成需求绑定/解绑操作，且操作后关联需求标签在 1 秒内同步更新到表格视图
- **SC-005**: 思维导图视图在包含 100 个节点时，首次渲染时间不超过 2 秒，节点交互响应时间不超过 300ms
- **SC-006**: 删除条目（软删除）后，其版本履历和关联需求关系在后台永久留存，管理员可在归档页面检索

## Assumptions

- 工作台上下文版本信息在创建/编辑时始终可用，若为空则默认使用系统当前版本号
- 需求池（需求文档列表）已在其他模块中维护，本功能仅通过 `demand_id` 引用，不管理需求池本身
- 思维导图渲染引擎（如 D3.js、AntV G6 等）已集成或可由项目技术栈支持
- 用户权限控制（谁可以编辑/删除/复制）沿用现有权限体系，本功能不做额外限制
- 版本号格式遵循语义化版本规范（如 `1.0.0`），由工作台上下文统一管理
- 软删除后的数据归档检索功能在后续迭代中实现，本功能仅确保数据不物理删除

## Scope & Boundaries

### In Scope

- 模块、功能、参数三类条目的版本溯源字段增改
- 表格视图列结构调整与版本履历悬浮浮窗
- 需求关联的增删查操作（含弹窗交互）
- 思维导图只读浏览与节点详情弹窗
- 新增/编辑弹窗的统一逻辑（含创建版本、需求关联）
- 存量数据兼容迁移方案
- 操作列统一按钮与权限控制

### Out of Scope

- 需求池本身的 CRUD 管理（假设已有独立模块）
- 版本号本身的生成规则与管理（假设由工作台上下文提供）
- 软删除数据的归档检索 UI（仅确保数据留存，UI 后续迭代）
- 跨项目数据隔离与权限控制（沿用现有体系）
- 思维导图的编辑功能（仅只读浏览）

## Dependencies

- 工作台上下文版本信息接口（获取当前版本号）
- 需求池数据接口（获取可关联的需求列表）
- 现有产品架构模块的数据库表结构（需扩展字段）
- 思维导图渲染组件/库

## Notes

- 本规格遵循项目 Constitution 中 "Version-Controlled Documentation" 原则，确保每个条目支持版本快照与追溯
- 数据层字段设计（`create_version`、`version_record`、`demand_relation`）需考虑后续扩展性，如增加 `operator_id` 等字段
- 思维导图视图的只读约束需在组件层面严格限制，避免用户误操作
