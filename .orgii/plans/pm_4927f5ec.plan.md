# PM 工作流平台 - 功能完善与打通计划

## 背景与现状

当前 PM 模块已具备雏形，包含：
- **14个业务模块**：需求、参数、测试用例、PRD、风险、竞品、路线图、会议、验收、FAQ、产品架构、原型、排期
- **基础功能**：项目创建/管理、条目 CRUD、富文本编辑、表格视图、甘特图、思维导图
- **版本系统**：条目级版本快照、分支、合并、对比、回滚（API + UI 组件已存在）
- **溯源面板**：基础信息展示（版本、来源、关联关系）
- **Agent 聊天**：意图识别、技能调用、LLM 对接

**但存在明显缺口**：
1. 各模块的基础信息字段与 PRD 规范不对齐（部分字段缺失/冗余）
2. 文件版本与基础信息未打通（版本快照未关联条目元数据）
3. 项目版本与文件版本信息孤立（项目版本 `pm_version` 表与条目版本 `pm_entry_version` 无关联）
4. 溯源功能仅展示静态信息，未关联版本流转链路，无合理性判定
5. Agent 功能仅实现基础聊天，未对接 OpenWebUI 原生 Agent/工具系统
6. 整体功能停留在"能用"阶段，未形成完整闭环

## 技术栈

- **前端**：Svelte 5 + SvelteKit + TailwindCSS + TipTap
- **后端**：FastAPI + SQLAlchemy + SQLite
- **AI**：OpenWebUI 原生 LLM 基础设施 (`generate_chat_completion`)

## 实施阶段

### Phase 1: 模块基础信息对齐（1-2 天）

**目标**：让每个模块的字段与 PRD 规范完全一致。

1. **梳理 PRD 规范字段**
   - 读取 `docs/prd/PM-Workflow-Platform-PRD-v1.0.md` 中各模块的数据模型定义
   - 对比当前 `src/routes/(app)/pm/[projectId]/[module]/+page.svelte` 中 `moduleConfig` 和表单字段
   - 列出差异清单

2. **补齐缺失字段**
   - 需求模块：补充 `userRole`, `expectedBenefit`, `relatedModules`, `tags`
   - 参数模块：补充 `required`, `description`
   - 测试用例：补充 `precondition`, `steps`, `inputData`, `expectedResult`
   - 风险模块：补充 `deadline`
   - 竞品模块：补充 `dimensions` 完整结构（已部分实现）
   - 路线图：补充 `layout`
   - 会议：补充 `participants`, `meetingDate`, `conclusions`, `actionItems`
   - 验收：补充 `passedItems`
   - FAQ：已完整
   - 产品架构：已完整

3. **统一字段存储格式**
   - 所有模块专属字段统一存入 `data` JSON 字段
   - 确保前端 `getEntryData()` 能正确读取
   - 更新创建/编辑表单，支持新字段

**关键文件**：
- `src/routes/(app)/pm/[projectId]/[module]/+page.svelte` — 模块配置和表单
- `src/lib/apis/pm/types.ts` — 类型定义补充

### Phase 2: 版本系统打通（2-3 天）

**目标**：项目版本 ↔ 文件版本 ↔ 基础信息 三层打通。

1. **项目版本与条目版本关联**
   - 在 `pm_entry_version` 表中增加 `project_version_id` 字段
   - 创建条目版本时，自动关联当前项目版本
   - 在版本列表中显示"所属项目版本"

2. **版本快照包含基础信息**
   - 创建版本时，快照不仅保存 `content`，还要保存完整的 `data`（含所有基础信息字段）
   - 确保回滚时基础信息一并恢复

3. **版本对比增强**
   - `PMVersionComparePanel` 当前仅对比 `content`
   - 增加 `data` 字段的对比（JSON diff）
   - 高亮显示基础信息变更

4. **版本筛选与追溯**
   - 条目列表支持按项目版本筛选
   - 在溯源面板中显示"创建于版本 X"、"最后变更于版本 Y"

**关键文件**：
- `backend/open_webui/models/pm.py` — 表结构 + 模型方法
- `backend/open_webui/routers/pm.py` — 版本 API 调整
- `src/lib/components/pm/PMVersionComparePanel.svelte` — 对比逻辑增强
- `src/lib/components/pm/PMVersionHistoryDropdown.svelte` — 版本关联展示

### Phase 3: 溯源功能完善（2-3 天）

**目标**：溯源关联版本流转，并能判定合理性。

1. **溯源数据模型完善**
   - `pm_entity` 和 `pm_relation` 表已存在，但 API 未完整暴露
   - 补充 `entity` 的 CRUD API（当前只有基础查询）
   - 补充 `relation` 的确认/删除 API

2. **自动创建溯源关系**
   - 当条目创建时，自动创建对应的 `pm_entity` 记录
   - 当条目关联其他条目（如测试用例关联需求）时，自动创建 `pm_relation`
   - 保存版本时，记录 `version_id` 到 relation 中

3. **溯源面板增强**
   - 当前 `PMTraceabilityGraph` 是静态展示
   - 接入真实 `pm_relation` 数据
   - 展示版本流转链路："v1.0 → 修改 → v1.1 → 关联需求 #123"

4. **合理性判定**
   - 实现简单的规则引擎：
     - 需求状态为 `approved` 但无关联测试用例 → 警告
     - 参数未关联 PRD 来源 → 提示
     - 风险概率 `high` 但无应对措施 → 警告
     - 版本回滚后关联关系断裂 → 提示
   - 在溯源面板中用颜色/图标标记合理性状态

**关键文件**：
- `backend/open_webui/routers/pm.py` — 补充 entity/relation API
- `src/lib/components/pm/PMTraceabilityGraph.svelte` — 图数据接入
- `src/lib/components/pm/PMTraceDetailPanel.svelte` — 详情面板增强
- `src/routes/(app)/pm/[projectId]/[module]/+page.svelte` — 自动创建关系逻辑

### Phase 4: Agent 功能对接 OpenWebUI（2-3 天）

**目标**：复用 OpenWebUI 原生 Agent/工具能力，而非独立实现。

1. **调研 OpenWebUI Agent 能力**
   - 参考 `https://docs.openwebui.com/` 了解 Agent/工具/函数调用机制
   - 分析 OpenWebUI 现有 `functions`, `tools`, `skills` 系统的区别
   - 确定最佳接入点：
     - **Functions**: 前端可调用的 JS 函数（适合 UI 操作）
     - **Tools**: 后端可调用的 Python 工具（适合数据处理）
     - **Skills**: 系统级能力（适合工作流）

2. **创建 PM 通用工具模块**
   - 设计通用接口：`PMTool<TInput, TOutput>`
   - 实现基础工具：
     - `EntryCreateTool` — 创建条目
     - `EntryUpdateTool` — 更新条目
     - `EntryQueryTool` — 查询条目
     - `VersionCreateTool` — 创建版本
     - `RelationCreateTool` — 创建关联
     - `TraceabilityAnalyzeTool` — 溯源分析
   - 每个工具封装为 OpenWebUI 兼容的格式

3. **对接 OpenWebUI Agent**
   - 复用 OpenWebUI 的 `generate_chat_completion` 基础设施
   - 将 PM 工具注册到 OpenWebUI 的工具系统中
   - 让 LLM 能自动调用 PM 工具完成工作流

4. **前端 Agent 面板增强**
   - `PMAgentChatPanel` 当前是独立实现
   - 改为调用 OpenWebUI 原生聊天 API，但注入 PM 上下文
   - 支持 Agent 建议的操作一键执行（调用 PM 工具）

**关键文件**：
- `backend/open_webui/pm/tools/` — 新建工具目录
- `backend/open_webui/pm/skills/` — 扩展现有技能
- `src/lib/components/pm/PMAgentChatPanel.svelte` — 对接原生 Agent

### Phase 5: 工作流整合与完善（2-3 天）

**目标**：基于通用模块快速开发工作流，完善整体闭环。

2. **版本管理页面完善**
   - 当前 `src/routes/(app)/pm/[projectId]/versions/+page.svelte` 为空
   - 实现项目版本列表、创建、切换

3. **溯源页面完善**
   - 当前 `src/routes/(app)/pm/[projectId]/traceability/+page.svelte` 为空

## 风险与待确认事项

1. **OpenWebUI Agent 接入方式**：需先调研确认 OpenWebUI 的 Functions/Tools/Skills 哪个最适合 PM 工具注册
2. **数据库迁移**：新增字段需要 Alembic 迁移脚本
3. **性能影响**：溯源关系自动创建可能增加条目操作延迟，考虑异步处理
4. **版本数据量**：条目版本快照频繁可能产生大量数据，需考虑清理策略
