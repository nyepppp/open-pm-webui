# PM模块完善实施计划

## Context

项目：OpenWebUI的PM工作流平台扩展（D:\project\pm-workspace\open-pm-webui）
技术栈：SvelteKit前端 + FastAPI后端 + SQLAlchemy/SQLite
当前状态：基础CRUD已有雏形，但核心功能未完善（所有模块共用相同表单，无差异化结构；版本API有但前端无面板；溯源关系表未建；Agent仅2个skill有实现）

PRD文档：docs/prd/PM-Workflow-Platform-PRD-v1.0.md、DB-Design-v1.0.md

## Approach

**目标**：让每个模块拥有符合PRD的差异化数据结构和编辑器

| 任务ID | 任务 | 负责人 | 产出物 | 验收标准 |
|--------|------|--------|--------|----------|

| 1.2 | 为13个已有模块扩展差异化字段（PRD的sections、参数的param_type/data_type、测试用例的steps/precondition、竞品的analysis维度、风险的probability/impact矩阵等） | sde-implementer | 模型更新+API更新 | 每个模块的API返回字段包含差异化结构 |

## 验收标准（整体）

| # | 验收项 | 标准 |
|---|--------|------|
| 1 | 手动可用 | 关闭AI后，所有功能100%可用 |
| 2 | 模块差异化 | 13个模块各自拥有符合PRD的字段结构和编辑器 |
| 3 | 版本管理 | 支持创建、对比、回滚条目版本；项目版本关联文件版本 |
| 4 | 关系追溯 | 支持双向追溯和影响分析，关系关联版本信息 |
| 5 | Agent增强 | 至少6个skill可用（PRD生成、需求分析、竞品调研、原型走查、参数提取、测试用例生成） |
| 6 | 数据互通 | 模块间可跳转，工作台数据真实，支持导入导出 |

## 风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| PRD中部分表字段与实际技术栈（SvelteKit+FastAPI）不完全匹配 | 中 | 阶段1先聚焦核心差异化字段，非核心字段可延后 |
| Agent skill调用OpenWebUI现有API可能存在权限/上下文隔离问题 | 中 | 4.1方案设计阶段充分调研，必要时独立封装 |
| Diff对比和版本回滚涉及数据一致性，可能引入bug | 高 | 2.3/2.4增加事务保护，回滚前强制备份 |
| 13个模块全部差异化开发工作量大 | 高 | 优先核心模块（PRD/参数/测试用例/竞品），其余可分批 |
| 关系网络查询性能（递归影响分析） | 中 | 3.4限制递归深度，增加缓存 |

## 建议执行顺序

1. **第一批（1~2周）**：1.1 + 1.2 + 核心模块1.3（PRD/参数/测试用例/竞品/风险） + 2.1 + 2.2
2. **第二批（2~3周）**：剩余模块1.3 + 1.4 + 2.3 + 2.4 + 2.5 + 3.1 + 3.2 + 3.3
3. **第三批（2周）**：3.4 + 3.5 + 3.6 + 4.1 + 4.2 + 核心skill（4.3~4.6）
4. **第四批（1~2周）**：剩余skill（4.7~4.10） + 5.1~5.5
