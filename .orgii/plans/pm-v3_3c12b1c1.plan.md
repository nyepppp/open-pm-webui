
# PM 工作台 v3 增强 — 版本·编辑器·批注·溯源·功能校验

## Approach

### A. 版本跟踪全局化 — 筛选/排序/分页/溯源联动

**现状问题**：条目列表只有搜索框，无法按版本号筛选、排序、分页；溯源功能未跟踪版本变更。

**步骤**：
1. **条目列表增加筛选栏**：在 `[module]/+page.svelte` 的搜索区域下方增加筛选条：
   - 版本号筛选（下拉/输入，支持 "无版本" / "v1.x" / 自定义）
   - 状态筛选（draft/review/approved/archived）
   - 优先级筛选（P0-P3）
   - 来源筛选（manual/excel/agent，仅 requirement）
2. **排序支持**：列表头增加排序切换（按 updatedAt / createdAt / priority / versionNumber 排序，升降切换）
3. **分页**：引入前端分页（pageSize=20），显示页码导航。当前 `entries` 数组全量加载，分页在前端 `filteredEntries` 上切片即可
4. **溯源跟踪版本**：Relation 类型增加 `versionSnapshot` 字段，溯源图节点显示版本徽章，连线详情面板展示关联时的版本快照。溯源图筛选器增加版本维度
5. **表格视图增加版本列**：所有 table 类型模块的 tableColumns 统一追加 versionNumber 列（当前只有 roadmap 有 versionId 列）

### B. 富文本编辑器定制 + 版本系统重构

**现状问题**：PMRichEditor 是通用 TipTap 封装，缺少 PM 专属版本控制逻辑、批注功能；版本比较/合并面板为占位符。

**步骤**：

#### B1. 编辑器版本系统
1. **EntryAnnotation 类型定义**（types.ts 新增）：
   ```ts
   interface EntryAnnotation {
     id: string;
     entryId: string;
     entryVersionId: string;
     textRange: { from: number; to: number };
     selectedText: string;
     content: string;
     highlightColor: string; // 默认黄色
     createdBy: string;
     createdAt: number;
     updatedAt: number;
   }
   ```
2. **手动保存 vs 自动保存**：
   - 编辑器增加 `autoSaveTimer`（30s debounce），自动保存时版本号不变（更新 content + updatedAt，不创建新 EntryVersion）
   - 用户点击"保存"按钮时，弹出确认框"是否产生新版本？是 → 递增版本号创建 EntryVersion；否 → 仅保存内容"
   - 编辑器头部显示当前版本号 + 保存状态（已保存/未保存更改/自动保存于 HH:MM）
3. **版本时间戳**：EntryVersion.createdAt 使用 ISO 8601 时间戳，UI 显示格式 `YYYY-MM-DD HH:mm`

#### B2. 版本比较 — 并排展示 + 红绿 diff
4. **重写 PMVersionComparePanel.svelte**：
   - 并排双栏布局（左=旧版，右=新版），非 overlay
   - 内容 diff 算法：将 HTML/TipTap JSON 按行拆分，使用 Myers diff 算法计算增删
   - 增加行 → 绿色背景 + 左侧 gutter 显示 `+`
   - 删除行 → 红色背景 + 左侧 gutter 显示 `-`
   - 未变更行 → 正常显示 + 行号
   - 行号对齐：左右两侧同步滚动
   - 结构化数据（metadata）差异以表格形式展示在底部

#### B3. 版本合并 — 行级/多行选择
5. **重写 PMVersionMergePanel.svelte**：
   - 三栏布局：左=源分支、中=合并结果、右=目标分支
   - 冲突行高亮显示，提供逐行选择按钮（"采用源"/"采用目标"/"手动编辑"）
   - 支持多行批量选择（Shift+点击 或 勾选框）
   - 合并确认后创建新 EntryVersion，branchName='main'

#### B4. 批注功能
6. **TipTap 批注扩展**（新建 `pmAnnotationExtension.ts`）：
   - 自定义 TipTap Mark：`annotation`，属性 { id, color }
   - 用户选中文本 → 浮动工具栏出现"批注"按钮 → 弹出输入框 → 标黄高亮
   - 批注列表面板：右侧侧栏显示当前文档所有批注，点击跳转定位
   - 批注数据存储在 `entry.data.annotations` 数组中
7. **AI 批注修改**：
   - 批注项增加"AI 修改"按钮
   - 点击后调用 OpenWebUI chat API（复用 agentChatStore.sendViaOpenWebUI），prompt 包含选中文本+批注内容+"请优化这段内容"
   - AI 返回结果插入编辑器（替换选中文本），同时保留批注标记

### C. PRD 文档功能同步 & 整体校验

**现状问题**：v2 spec 中部分功能与当前实现偏差较大，需要对照 PRD 文档做全面校验。

**步骤**：
1. **功能差距审计**：逐项对照 spec-v2.md 的 FR-001 到 FR-027，标记已实现/部分实现/未实现
2. **溯源校验**：
   - 溯源图连线创建是否真正调用了后端 API 创建 Relation？验证 PMRelationTypeSelector → relation.create 流程
   - 溯源图节点点击是否弹出详情面板？验证 PMTraceDetailPanel 数据加载
   - 右键菜单、双击跳转是否完整实现？
3. **版本系统校验**：
   - PMVersionHistoryDropdown 是否在所有模块的条目卡片上渲染？当前只在富文本卡片视图有，表格视图行缺少版本徽章
   - 自动版本快照（saveEntryDoc 中的 createEntryVersion）是否可靠执行？
4. **富文本统一校验**：所有 rich 类型模块（prd, meeting）是否使用同一个 PMRichEditor？FAQ 答案字段是否已切换为富文本？
5. **测试用例**：为上述校验项编写前端可执行的手动测试清单，标记 pass/fail

### D. 溯源增强 — 版本感知 + 筛选

**步骤**：

2. 溯源图顶部增加筛选栏：按模块类型、版本号、关系类型筛选

## Risks & Open Questions

1. **TipTap 批注扩展复杂度**：自定义 Mark 需要处理选区冲突（批注 vs 其他格式化标记交叉），可能需要使用 TipTap 的 Collaboration 扩展或自定义 NodeView。建议先实现基础版（单层高亮+弹窗），后续迭代支持嵌套批注。

2. **Diff 算法精度**：Myers diff 对富文本 HTML 的行级对比可能产生大量噪声（HTML 标签变化 vs 内容变化）。建议：先 strip HTML 标签做纯文本 diff，再映射回 HTML 位置做高亮。TipTap JSON 级别的 diff 更精确但复杂度更高。

3. **自动保存与手动保存的竞态**：用户编辑中触发自动保存，同时用户点击手动保存，需确保不会创建重复版本。方案：手动保存时取消 pending 的自动保存 timer。

4. **后端 API 限制**：当前前端直接调用 OpenWebUI 的通用 API（notes/memories），PM 数据实际存储在 `pm_module_entries` 表中。批注、版本等新字段需要后端支持。如果后端未升级，前端方案：将批注数据存在 `entry.data.annotations`（JSON 字段），版本快照存在 `entry.data` 中。

5. **分页策略**：当前前端全量加载，如果条目超过 500 条可能卡顿。建议先用前端分页，后续接入后端分页 API。

6. **spec-v3 与 spec-v2 的关系**：建议在现有 `specs/001-pm-workspace-redesign/` 目录下新增 `spec-v3.md` 和 `plan-v3.md`，复用已有的 feature directory。
